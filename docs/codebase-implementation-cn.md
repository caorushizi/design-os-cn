# 代码库实现

从 Design OS 导出您的设计后，您就拥有了一个准备好进行实现的完整移交包（handoff package）。本指南涵盖了如何与您的 AI 编码 Agent 合作来构建产品。

## 快速入门

1. 将 `product-plan/` 文件夹复制到您的目标代码库中
2. 启动您的 AI 编码 Agent (Claude Code, Cursor 等)
3. 选择您的实现方法：一次性（one-shot）或逐板块（section-by-section）

## 实现方法

### 选项 A：增量实现（推荐）

适用于较大的产品，或者当您希望逐步审查进度时。

**工作原理：**

按顺序执行说明文件：

1. **外壳 (Shell)** (`instructions/incremental/01-shell.md`) — 设计标记（Design tokens）和应用程序外壳
2. **板块 (Sections)** (`instructions/incremental/02-*.md`, `03-*.md` 等) — 每个功能板块，一次一个

对于每个里程碑：

1. 打开 `product-plan/prompts/section-prompt.md`
2. 填写顶部的变量参数 (SECTION_NAME, SECTION_ID, NN)
3. 添加任何针对特定板块的注释
4. 将提示词 (prompt) 复制/粘贴到您的编码 Agent 中
5. 回答澄清性问题，并让 Agent 开始实现
6. 在进入下一个里程碑之前进行审查和测试

**板块提示词 (section prompt)：**

- 引用了该板块的说明文件和资产
- 指向 `tests.md` 以获取 UI 行为测试规范
- 引导 Agent 在实现之前提出澄清问题

### 选项 B：一次性实现 (One-Shot Implementation)

适用于较简单的产品，或者当您希望在一个会话中构建所有内容时。

**工作原理：**

1. 打开 `product-plan/prompts/one-shot-prompt.md`
2. 添加任何额外的注意事项（技术栈偏好，约束条件）
3. 将提示词复制/粘贴到您的编码 Agent 中
4. 回答 Agent 关于认证、用户建模等方面的澄清问题
5. 让 Agent 规划并实现所有内容

提示词引用了 `product-overview.md` 和 `instructions/one-shot-instructions.md`，并引导您的 Agent 在开始之前审查所有提供的文件并提出澄清问题。

## 测试驱动开发 (TDD)

每个板块都包含一个 `tests.md` 文件，其中包含详细的测试编写说明。我们推荐 TDD（测试驱动开发）方法：

1. **阅读测试说明** — 审查 `sections/[section-id]/tests.md`
2. **编写失败的测试** — 根据描述的用户流程和断言
3. **实现功能** — 使测试通过
4. **重构** — 清理代码，同时保持测试全绿（通过）

测试说明包括：

- **用户流程测试 (User flow tests)** — 关键交互的成功和失败路径
- **空状态测试 (Empty state tests)** — 验证不存在任何记录时的行为
- **组件交互测试 (Component interaction tests)** — 特定的 UI 元素和行为
- **边缘用例 (Edge cases)** — 边界条件和状态过渡

测试说明是**与框架无关的 (framework-agnostic)** —— 它们描述了要测试**什么**（WHAT），而不是**如何**（HOW）测试。将其调整为您的测试设置（Jest, Vitest, Playwright, Cypress, RSpec, Minitest, PHPUnit 等）。

## 规范驱动开发 (Spec-Driven Development)

我们还推荐一种规范驱动的方法：

1. **审查设计** — 了解设计了什么以及为什么
2. **提出澄清问题** — 在编码前消除任何歧义
3. **规划实现** — 决定架构、数据层和集成方法
4. **首先编写测试** — 基于提供的测试说明
5. **实现** — 构建代码以使测试通过
6. **验证** — 确保实现与设计相匹配

这种方法可以防止由于误解而浪费工作，并确保正确的实现能支撑好 UI 设计。

## 澄清问题

在最终确定任何实施计划之前，鼓励您的 Agent 审阅所有提供的文件，并就以下问题提出澄清：

- 您的技术栈和任何现有的代码库约定
- 身份验证（Auth）和用户管理的方法
- 如何为您的后端需求扩展数据形状（data shapes）
- 任何需要澄清的产品需求
- 在实现之前所需的任何其他内容

## 包含的内容与您要构建的内容

Design OS 的导出提供：

- **已完成的 UI 组件** — 基于 props，完全格式化并加了样式，响应式，支持暗色模式
- **产品需求** — 规范（Specs），用户流程和范围定义
- **设计系统标记 (tokens)** — 颜色，排版字体，CSS 自定义属性
- **示例数据** — 展示组件期望获得的数据的形状
- **UI 行为测试规范** — 从用户的角度看要测试什么

您的实现 Agent 将构建所有其他内容 —— 后端、数据层、路由、状态管理和测试。组件通过 props 接受数据并触发回调；您如何履行这些契约是一项实现层面的决策。

**UI 组件是完整且可用于生产环境的。** 把它们连接组装起来，不要重新构建它们。

## 实用提示

- **使用预先编写好的提示词** — 它们会引导您的 Agent 审查设计并提出正确的问题
- **始终包含 product-overview.md** — 它提供有关整个产品的基本上下文
- **首先编写测试** — 使用 `tests.md` 说明进行 TDD
- **增量审查** — 逐个板块地实现，可让您及早发现问题
- **先使用示例数据进行测试** — 在构建真实的 API 之前，请使用提供的 sample-data.json
- **处理空状态** — 确保在没有记录时（如首次使用的用户）有良好的用户体验
- **信任这些组件** — 它们已经被设计和赋予样式了；去连接组装它们，而不是重新构建
