# 什么是 Design OS？

Design OS 是您的产品想法与代码库之间缺失的那个设计步骤。

它是一个产品规划和设计工具，它可以帮助您定义产品愿景，勾勒数据形状（data shape），设计 UI，并导出可用于生产环境（production-ready）的组件以供实现。无需直接开始编码，您可以通过一个有向导的流程来确定您要构建什么以及为什么构建，然后将所有内容移交给您的编码 Agent 以正确构建它。

## 问题所在

AI 编码工具在快速构建方面表现出色。但结果往往不尽如人意。您描述了您想要的，Agent 构建了*一些东西*，但那不是您设想的。UI 看起来很普通。功能只实现了一半。您花在修复和重定向上的时间甚至与直接构建所花的时间一样多。

核心问题：我们要求编码 Agent在*同时*弄清楚要构建什么以及去构建它。设计决策是在运行中做出的，深埋在代码中，如果不重新开始，就不可能调整。没有规范（spec）。没有共识。没有什么是“完成”的单一事实来源。

## Design OS 流程

Design OS 提供并支持一个引导式的设计和架构流程。您与 AI 合作，通过结构化的步骤逐步完成：

1. **产品规划** — 定义您的愿景，分解您的路线图，并对您的数据进行建模
2. **设计系统** — 选择颜色、排版，并设计您的应用外壳结构（application shell）
3. **板块设计（Section Design）** — 针对每个功能区域：指定要求，生成示例数据，并设计屏幕（界面）
4. **导出** — 生成完整的移交包以供后续实现

每一步都是一次对话。AI 会提出问题，您提供方向，然后一起塑造一个符合您愿景的产品——在任何代码实现开始之前。

## 它是如何工作的

Design OS 本身是一个独立的代码库——这是一个您在实际构建*之前*使用的设计环境。完成设计后，您将导出组件和资源，以便导入到您实际产品的代码库中。

您通过斜杠命令 (slash commands) 与 Design OS 交互。每个命令都会引导您完成该流程的特定部分：

- `/product-vision` — 定义您的产品，将其拆分为各个板块，并勾勒出数据形状 —— 都在一个连贯的流程中完成
- `/design-tokens` — 选择您的调色板（color palette）和字体排版（typography）
- `/design-shell` — 设计导航和整体布局
- `/shape-section` — 定义一个板块的范围、需求，并生成示例数据和类型（types）
- `/design-screen` — 创建可用于生产环境（production-ready）的 React 组件
- `/screenshot-design` — 截取已完成的设计界面屏幕截图，以便快速提供视觉参考
- `/export-product` — 生成完整的移交打包（handoff package）
- `/product-roadmap` — 更新您的产品板块计划（在初始创建之后使用）
- `/data-shape` — 更新您的数据实体（在初始创建之后使用）
- `/sample-data` — 更新示例数据和类型（在初始创建之后使用）

关于完整的工作流请参阅 [使用方法 (Usage)](usage.md)。

## 它是为谁准备的

- **技术构建者 (Technical builders)** — 能够完全掌控架构和设计决策，而无需自己完成所有跑腿的繁琐工作。
- **非技术型产品构建者 (Non-technical product builders)** — 对产品应该如何运作有强烈的愿景，并具备在系统上留下自己印记的能力——这不需要您有一定的技术背景。

## 兼容性

- 可与任何 AI 编码 Agent 配合使用：Claude Code、Cursor、Copilot 或任何可以从交接文档（handoff）实施代码构建的工具
- 您的前端需要使用 React 和 Tailwind CSS
- 您的后端可以是任何技术栈——Rails、Laravel、Next.js、Python，任何都可以

## 开源

Design OS 是免费、开源并在本地运行的。
