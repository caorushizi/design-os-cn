# 更新日志

## [2026.2.9]

- 简化了 `/product-vision` : 现在通过一次对话式流程即可生成产品概述、产品路线图和数据形状。在回答澄清问题之后，所有三个文件都会自动写入，而不需要草稿审批。
- 更新了 `/product-roadmap` 和 `/data-shape` 以遵循更新模式：如果文件已存在，则显示当前状态并询问需要更改什么；如果不存在，则根据前置条件立即生成。
- 简化了 `/shape-section` 流程：在回答澄清问题后，规范（spec）、示例数据和 TypeScript 类型现在会自动生成，无需草稿审批。
- 简化了 `/sample-data` 流程：当独立运行时，立即自动生成数据。当数据已经存在时，询问需要更改的内容，然后自动更新。
- 重新定义 `/export-product` 作为 UI 设计移交（而不是全栈规范）。从前言、里程碑说明、提示词模板（prompts）和测试用例中删除了规定性的（prescriptive）后端/数据库指南。移交现在专注于 UI 组件、产品需求和用户流程——将后端架构决策留给负责实现的 agent。
- 将 foundation + shell 合并到单个 `01-shell.md` 里程碑（设计标记（design tokens）+ 应用程序外壳）。各个主要板块（Sections）现在从里程碑 02 开始。
- 在导出包中将 `data-model/` 重命名为 `data-shapes/`，并新增了 `overview.ts`，它汇总了所有板块的类型（types）作为统一参考。
- 简化了导出提示词（prompts）文件 —— 用更简短、开放式的澄清提问，取代了原本规定性的（prescriptive）后端问题分类。
- 重新聚焦导出测试规范（`tests.md`）仅在 UI 行为上 —— 删除了后端测试指导。
- 把项目各个地方的 `/data-model` 改名为了 `/data-shape`。命令、UI、路由和所有的相关引用现在全部使用“数据形状（data shape）”，重在强调这一步是为了勾勒数据的基本轮廓，而非定义一个最终版本的数据模型。
- 切换为基于日期的版本号分配方式。
