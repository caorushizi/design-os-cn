# 数据形状 (Data Shape)

您正在协助用户创建或更新他们产品数据的总体形状 —— 核心实体（"名词"）以及它们之间如何相互关联。这创建了一个共享的词汇表，从而确保在生成样本数据和各板块界面设计时保持一致性。这并非最终的数据模型 —— 它只是一个供实现 Agent （编码助手）后续去扩展和完善的一个起点。

## 第 1 步：检查当前状态

首先，检查 `/product/data-shape/data-shape.md` 是否存在。

---

## 如果数据形状已存在 (更新流程)

阅读以下文件：
- `/product/data-shape/data-shape.md`
- `/product/product-overview.md` (如果存在的话，用以了解上下文)
- `/product/product-roadmap.md` (如果存在的话，用以了解上下文)

展示当前状态并询问需要更改什么：

"您当前的数据形状定义了以下实体：

- **[实体1]** — [描述]
- **[实体2]** — [描述]

**关连关系：**
- [关系 1]
- [关系 2]

您想对这些实体或他们间的关系做哪些更改？"

等待用户回复说明他们想要更改的内容。收到他们的意见后，**立即继续**根据他们要求的更改更新 `product/data-shape/data-shape.md` —— 不要先展示草稿来征求批准。

更新后，通知用户：

"我已经根据您的反馈更新了数据形状。您可以查看所做的更改，如果还需要进一步调整，请告诉我。"

到此停止 —— 以下剩余的步骤是针对从头开始生成新的数据形状的情况。

---

## 如果目前不存在数据形状 (全新创建流程)

### 检查前置条件

阅读：
1. `/product/product-overview.md` 以了解这个产品是做什么的
2. `/product/product-roadmap.md` 以了解计划中的各个板块(sections)

如果有任意一个文件缺失，则告知用户：

"在定义数据形状之前，您需要先确立产品愿景。请先运行 `/product-vision` 命令。"

若不满足前置条件则停在这步。

### 分析并生成

审查产品概述和路线图，然后**立即继续**直接创建数据形状文件 —— 不要先展现一份草稿来期许经核准再动作。

识别出：
- **实体名称 (Entity names)** — 主要的名词（也就是用户创建、查看或管理的那些事物主体）
- **通俗易懂的描述 (Plain-language descriptions)** — 每个实体所代表具体的含义是什么
- **关系 (Relationships)** — 实体之间是如何连接的

以如下格式创建 `/product/data-shape/data-shape.md`：

```markdown
# 数据形状

## 实体

### [实体名称]
[用通俗的大白话描述这个实体到底表示什么，和它在这个系统当中起到的用途。]

### [另一个实体]
[大白话描述。]

[按需添加更多实体]

## 关系关联

- [实体1] 拥有多个 [实体2] (has many)
- [实体2] 隶属于 [实体1] (belongs to)
- [实体3] 隶属于 [实体1] 同时又隶属于 [实体2]
[按需添加更多关系联系]
```

### 确认完成

"我已经为您在 `/product/data-shape/data-shape.md` 下创建好了数据形状。

**已定义的实体：**
- [列出各实体]

**关系关联：**
- [列出主要关系]

这为您之后勾勒屏幕设计界定出了一套共享通用的词汇表。在您接下来运行 `/sample-data` 指令时，它将引述参考这些名词实体来确保在诸多不同下属模块之间维系命名使用上的一致。

您可以去查阅一下看看有没啥要调改的随时说一声。一旦你备好了，您敲击 `/design-tokens` 即可移步下一阶段进行色板定调和选拔合适配伍的字体款型来组建设计标记(design tokens)。"

## 重要说明

- 尽量保持**极简 (minimal)** —— 只需有实体名字、附带上的说明描述及互相连结的从属干系
- 绝对不要（Do NOT）去细化构建详细深入的字段排布胸案 (schema) 或者参数体位和校验规章等
- 用即使欠缺 IT 技术底细的人也轻易识得的平白直说去娓娓道来
- 关系就是纯图解意识层面上的指代 —— 它们只是陈述指代这些业务资料站在消费客户的角度视界里如何关联相接，决然非是要搭建数据库里的表外键底层结构
- 真正到了去具体规划它那骨架模型、落地怎么储备以及该再怎样增宽扩源延展这类实体对象，这些细活儿那是留待后期分包给实现建设期代码 Agent （implementation agent）的义务职责
- 各实体的定名都该规矩到单数表达（单数如 User, Invoice, Project — 非复数为 Users, Invoices）
- 决绝不能呈上一个初拟的草图方案等着别人拍板示可（Do NOT present a draft for approval） —— 痛快点立即生造直接产生此文件之后再留待后续的核查看阅
- 要是人客户过目完了还再另外发了指令说该这么那般的改去调换，立马言听计从利索地下手修改刷新这个文件
