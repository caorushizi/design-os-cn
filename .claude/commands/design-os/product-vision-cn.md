# 构筑产品愿景 (Product Vision)

您正在协助用户去确立其产品的蓝图愿景。整个流程是互动问答式的探讨过程，最终产出三份核心定纲文件：`product-overview.md`、`product-roadmap.md` 以及 `data-shape.md`。

## 第 1 步：收取初始的点子或素材

头件事，通过开放式的问题向他们讨要未加斧正的原生点子或素材：

"我很乐意为您打定这款产品的愿景框架。请将有关这款产品的各样杂念、手稿或想法与我说说。比如，它意图排解什么困难？受众群体是哪些人？别多虑结构或格式等繁文缛节，尽管放开讲就是了。"

话毕便等候其回应。

## 第 2 步：剥笋般地追问以清晰眉目

收揽过这些简短描述后，用 AskUserQuestion 这个神器抛出追问。要如同聊天侃大山那般去套取话头，每次问一至两点。

### 关于构画产品长相特有气质的诸般疑问

替其核心的属性做起勾勒圈划：

- **产品大名** - 给叫个好听易记也响当的名。
- **产品核心大义说明** (1-3 句扼要简话直戳其心髓)。
- **这产品是为攻克排遣化哪些痛点** (1-5大点去点出痛处的要害)。
- **又拿何其有术解脱化之解法方略** (给出定锤的去实化之方案解脱之道)。
- **为此这能有这些得要些啥本领功能点**

**重大提点：** 要是自始至终客官没给定指明一产品称号大名，务必一定必须要求个号名号为先：
- "可愿给指个好彩叫大名名头没有？(好记又精干些的)"

发问引导举例：
- "大抵主力会是谁在用？描描可好？"
- "您想去扫除解去的最为痛最巨最大痛难难点是什么？"
- "搁在这前，没有您的产品时大众是怎么应付解决此难题的？"
- "您家出路较他路之好在哪胜点何足道？"
- "拔尖出头必得有的三到五大立足功能有那几个？"

### 关乎路线版图划分之探问

为划区圈地立出各领业务功能之地与板块起问：

- "这此款这当中有什么大的业务功能区抑或界面？(诸如大表台 Dashboard，或者 Invoices)"
- "首冲在急之是要先开辟何处为先呢？"
- "又可有独立其外的功能板块？"

### 数据定大纲描其状之疑问

圈出这些要在其之中把管流动的核心事物 (名词实体 nouns)：

- "这里边让受众摆弄过目的核心事物都叫什么？(诸如 Projects, Clients 等)"
- "他们互相里是怎么牵缠关联的？"

目的只是获取足量信息来建立三文件，并不需要巨细无漏，只要核心实体及其关系即可。

## 第 3 步：跑过自发之动能 —— 取以造建立起三册本档

一旦凑齐足够材料，**不需要草图确认，直接写入三份文件**：

### 3a: 铸这第一本 - 产品总概况要录 (Product Overview)

`/product/product-overview.md` :

```markdown
# [产品名称]

## 概述
[最终锤定不加过1-3句简精的描述]

## 痛点与化解之道

### 痛点 1: [痛点称呼]
[它是何种手段用1-2两句解开之]

### 痛点 2: [痛点称呼]
[它解化之说明]

[需其还有余则加之，最多五 5点]

## 核心主打功能

- [功能 1]
- [功能 2]
- [功能 3]
[再有即添加]
```

### 3b: 建起去拿这一划版路地图之卷 (Product Roadmap)

`/product/product-roadmap.md` :

```markdown
# 产品路线指引

## 诸等模块

### 1. [第一模块抬头名]
[极尽一句话这仅在此说明]

### 2. [第二模块抬头名]
[一句话的简述说明]

### 3. [第三模块]
[一句话之说明]
```

注意: 使用带 `### 1. Title` 的数字格式是强制需要的以供识别解析（the numbered format is required for parsing）。

### 3c: 造个数据概略形廓本在之图 (Data Shape)

`/product/data-shape/data-shape.md`:

```markdown
# 数据外形 (Data Shape)

## 实体 (Entities)

### [实体名 (EntityName)]
[大白话解释此实体代指何物以及其作用]

### [第二个实体 (AnotherEntity)]
[大白话说明]

[需加则加]

## 联系关系 (Relationships)

- [实体 1] 拥有一众 (has many) [实体 2]
- [实体 2] 尽归于属 (belongs to) [实体 1]
- [实体 3] 既同归于 [实体 1] 与 [实体 2] (belongs to both)
[若需加也如法加之]
```

**须谨记**：重在说明它们代指什么而非详细列举内部每一字段 (not every field it contains)；并且命名需用单数形式 (User, Invoice, Project — not Users, Invoices)。

### 3d: 向用户通告禀报

待这三册造完完毕之后：

"我已经为您将 **[产品名称]** 的基础打好了：

1. **产品概览（Product Overview）** — `product/product-overview.md`
2. **路线图（Product Roadmap）** — `product/product-roadmap.md` ([N] 个板块)
3. **数据形体（Data Shape）** — `product/data-shape/data-shape.md` ([N] 个实体)

**各个分区板块有：**
1. **[板块名称 1]** — [极简述说一二]
2. **[板块名称 2]** — [同理简述之]
3. **[板块名称 3]** — [说明一二]

**其中包含的核心实体有：（Core Entities:）** [实体 1], [实体 2], [实体 3]

看看各位去读过看一眼这各文档。如果有想调整的可再知会我。当您准备好以后敲下去执行令 `/design-tokens` 即可去为产品定色彩与字等设计系；又或者是敲 `/shape-section` 去开始造一个其中单独版块的面孔图。"

## 重要事宜 (Important Notes)

- 谈吐如常非死僵般的话术进行
- 答案模糊则发声跟进追问确切
- 助其发散构想，不要仅仅作笔记抄写员
- 出产物果决纯青不可啰嗦
- 格式一点错不可须必契合，以供解析
- 绝对务定**此产品带有名字** —— 如果客并未给，记得求要之
- **不要给过什么草样求确认（Do NOT present a draft for approval）**
- 发发通告报备完毕之后遇有欲再有调更的须当时马上予给修正之。
